# 今日营养、训练记录与智能草稿设计规范

**状态：** 已确认，待实施计划  
**日期：** 2026-07-19  
**范围：** 今日页、用户目标、餐食记录、训练记录、本地目录、智能输入、Agent 草稿、旧数据迁移  
**不在范围：** 账户数据导出、食谱编辑、条码识别、图片识别、真实手机号或邮箱验证

## 1. 背景与目标

当前产品已经具备可运行的 MVP，但今日页、表单、统计和 Agent 行为仍保留明显的演示系统结构：

- 今日页同时承担概览、餐食表单、训练表单和智能输入，页面职责过多；
- 数值输入默认显示无法解释的 `0` 或 `00`，字段标签和单位不完整；
- 餐食缺少碳水和脂肪等完整数据，训练记录无法表达场次、动作和逐组信息；
- 智能输入可能直接写入正式数据，缺少草稿和用户确认边界；
- 现有 CSV 和文本知识库无法可靠支持版本化目标、历史快照、目录搜索和原子确认。

本阶段把系统升级为一个以记录为核心、可以持续使用和交付的产品。用户应当能够：

1. 完成档案和目标设置；
2. 查看某一天的营养与训练状态；
3. 通过目录搜索、结构化表单或智能输入记录餐食和训练；
4. 明确区分人工数据、设备数据、固定程序估算和 Agent 估算；
5. 在确认之前编辑草稿，确认后得到稳定、可追溯的历史记录。

本规范覆盖并替代以下旧文档中的冲突要求：

- `2026-07-09-today-first-page-hierarchy-design.md` 中“今日页内嵌完整录入表单”的要求；
- `2026-07-14-product-task-routes.md` 中直接基于 CSV 扩展智能草稿的实现方向；
- 任何允许 Agent 推荐或改变用户总体目标的描述。

Today-first 的顶层产品结构继续保留，但今日页改为概览与任务入口，完整录入进入独立路由。

## 2. 已确认的产品边界

### 2.1 固定程序与 Agent

系统存在两条互相解耦的处理路径：

- **固定程序路径：** 目标计算、营养换算、每日汇总、目录匹配、MET 消耗估算、字段校验和事务写入；
- **Agent 路径：** 对未知或含糊内容生成结构化建议草稿，说明假设、范围和来源。

固定程序在没有模型配置时仍须完整可用。Agent 不能直接新增、修改或删除已确认记录，也不能在模型不可用时用固定模板伪装成模型结果。

### 2.2 目标边界

目标分为两类：

- **总体目标：** 减脂、维持、增肌，只能由用户选择；
- **每日目标：** 热量、碳水、蛋白质、脂肪，可由固定程序生成、用户修改，或由 Agent 在用户主动请求时提供建议草案。

Agent 不推荐和不改变总体目标。所有 Agent 每日目标建议必须由用户确认。

### 2.3 记录边界

- 草稿可以不完整，不参与任何统计；
- 已确认记录必须满足完整性校验；
- 智能输入产生多条记录时，选中内容必须全成功或全回滚；
- 公共目录更新不能改变历史餐食或训练数据；
- 训练消耗不从营养摄入或每日营养目标中扣除。

## 3. 总体架构

采用 React、FastAPI 和 SQLite 组成的模块化单体。

### 3.1 分层

后端每个业务模块采用三层边界：

1. API 路由负责认证、请求解析和响应格式；
2. Service 负责业务规则、权限和事务；
3. Repository 负责 SQLite 访问。

Service 不直接拼接 SQL，API 路由不直接调用 Repository。Repository 接口保留未来迁移 PostgreSQL 的可能性，但本阶段不实现双数据库支持。

### 3.2 后端模块

| 模块 | 职责 |
| --- | --- |
| `profile` | 身体档案、能量估算参数、活动水平和安全限制 |
| `targets` | 总体目标、每日目标计算、版本和建议草稿 |
| `meals` | 餐次、食物条目、餐食确认和历史快照 |
| `workouts` | 训练场次、力量动作、训练组、有氧项目和消耗估算 |
| `catalog` | 公共及个人目录、别名、收藏、最近使用和搜索 |
| `drafts` | 自动保存、版本冲突、过期和原子确认 |
| `agent` | 模型适配、结构化输出校验和失败恢复 |
| `daily_summary` | 按日期聚合目标、营养和训练数据 |
| `migration` | 数据库版本、目录导入和旧 CSV 迁移 |

### 3.3 前端边界

前端按页面任务拆分，不再由 `Today.tsx` 承担所有表单状态。后端是业务数据的唯一事实来源，前端不独立维护长期统计结果。

## 4. 核心数据模型

所有用户业务表包含 `user_id`、`created_at` 和 `updated_at`。正式记录保存快照，不依赖目录当前值。

### 4.1 档案与目标

#### `user_profile_versions`

保存：

- 年龄，且只支持 18 岁及以上用户；
- 身高和体重；
- 能量估算参数：男性、女性、未指定/中性估算；
- 活动水平：久坐、轻度、中度、高度；
- 自动计算禁用条件。

能量估算参数只用于计算，不作为身份表达字段。

#### `overall_goal_versions`

保存减脂、维持或增肌及其生效时间。总体目标由用户选择，不接受 Agent 写入。

#### `daily_target_versions`

保存热量、碳水、蛋白质、脂肪、生效日期、来源、计算依据和公式版本。来源包括：

- `deterministic_calculation`；
- `manual`；
- `agent_confirmed`。

档案或总体目标变化只生成重算预览。用户确认后创建新版本，不覆盖历史日期使用的目标。

### 4.2 每日和餐食

#### `daily_logs`

按用户和日期唯一，保存当天计划餐数和目标快照引用。默认计划餐数为 3，但当天可以修改；只有用户明确选择“设为以后默认”时才更新未来默认值。

#### `meals`

保存稳定 ID、日期、名称、餐次类型和排序。系统不预创建空餐，只有添加食物后才显示餐次。

默认名称为早餐、午餐、晚餐；新增餐次使用加餐名称。用户可改显示名称，但后端稳定 ID 不变。向已有餐次增加食物不会增加餐次数，创建新餐次才会。

#### `meal_items`

每餐包含多个食物条目。条目保存：

- 食物名称和份量；
- 单位与每 `100g`、每 `100ml` 或每份基准；
- 热量、碳水、蛋白质和脂肪快照；
- 来源、估算标记、假设和不确定区间。

完整人工创建的自定义食物自动进入用户私有目录。Agent 估算条目只有在用户明确选择时才进入私有目录。

### 4.3 训练

#### `training_sessions`

训练以场次为聚合根。同一天允许多个场次，一个场次可同时包含力量和有氧项目。保存日期、标题、可选开始时间、可选时长和强度。

#### `strength_exercises` 与 `strength_sets`

力量动作必填：

- 动作；
- 主要肌群；
- 组数；
- 每组次数。

重量可选，并支持自重。目录动作提供主要和次要肌群；自定义动作必须填写主要肌群。界面默认使用“组数 x 次数 x 重量”，并允许展开成逐组记录。

#### `cardio_items`

有氧项目必填运动和时长。设备消耗可选；有设备值时优先使用，否则使用 MET 固定公式估算。

### 4.4 目录和草稿

公共目录和用户私有目录使用统一中间模型，并通过所有者和来源隔离。主要实体包括：

- `food_catalog`；
- `exercise_catalog`；
- `catalog_aliases`；
- `favorites`；
- `recent_catalog_usage`。

SQLite FTS5 支持中文、英文、别名和拼音检索。

`record_drafts` 保存草稿类型、结构化 JSON、Schema 版本、乐观锁版本、Agent 状态和过期时间。草稿默认保留 30 天。确认时在单个事务中转换为正式关系记录。

## 5. 固定计算规则

### 5.1 每日目标

基础代谢采用版本化的 Mifflin-St Jeor 公式：

```text
男性参数：BMR = 10W + 6.25H - 5A + 5
女性参数：BMR = 10W + 6.25H - 5A - 161
未指定：采用以上两个结果的中值
```

其中 `W` 为 kg、`H` 为 cm、`A` 为年龄。

活动系数：

| 活动水平 | 系数 |
| --- | ---: |
| 久坐 | 1.2 |
| 轻度 | 1.375 |
| 中度 | 1.55 |
| 高度 | 1.725 |

总体目标调整：

- 减脂：维持热量减 15%；
- 维持：不调整；
- 增肌：维持热量加 10%。

宏量营养素：

- 维持目标蛋白质：`1.6 g/kg`；
- 减脂或增肌蛋白质：`1.8 g/kg`；
- 脂肪：`0.8 g/kg`；
- 碳水：`(目标热量 - 蛋白质 x 4 - 脂肪 x 9) / 4`。

固定程序采用一致的取整策略，并保存公式版本。自动结果最低热量为：男性参数 1500 kcal、女性参数 1200 kcal、中性估算 1350 kcal。

以下情况禁用自动目标生成：孕期、哺乳期、进食障碍、相关肾脏或代谢疾病、其他相关医疗情况或处方饮食。产品只显示限制和就医提示，不提供医疗替代方案。

手动每日目标只接受以下闭区间：热量 `800-6000 kcal`、碳水 `0-1000 g`、蛋白质 `20-400 g`、脂肪 `10-300 g`。边界外直接拒绝；边界内相对固定程序基线偏差超过 20% 时要求二次确认。宏量营养素折算热量与总热量偏差超过 10% 时提示不一致，但允许用户确认保存。自动计算结果也不得超过这些硬边界。

### 5.2 训练消耗

无设备值时，有氧采用：

```text
消耗 kcal = MET x 3.5 x 体重kg / 200 x 分钟
```

力量训练只有填写时长后才根据低、中、高强度对应的有来源 MET 估算。所有固定程序计算的训练消耗显示“约”或“估算”，并保存体重快照、MET、时长、公式版本和来源。

### 5.3 食物换算

目录和自定义条目统一换算到每 `100g`、每 `100ml` 或每份。自定义食物必须同时具有计量基准、热量、碳水、蛋白质和脂肪。

热量与三大营养素折算值差异较大时提示，但考虑纤维、糖醇等因素，不自动否定可信公共数据。Agent 返回区间不能直接参与统计，用户必须确认具体值。

## 6. 页面与交互设计

### 6.1 今日页 `/today`

采用营养优先的纵向结构：

1. 日期与日历入口；
2. 热量、碳水、蛋白质、脂肪的已摄入、目标和剩余；
3. 餐食区域，显示“已记录 N / 计划 M 餐”，空餐不显示；
4. 当天存在训练时显示训练区域；
5. 添加餐食、添加训练和智能输入命令。

今日页负责聚合展示和导航，不内嵌大型录入或编辑窗口。

### 6.2 餐食录入 `/today/meal/new`

采用搜索优先、逐步展开：

1. 选择已有餐次或创建新餐次；
2. 搜索目录；
3. 选择食物并填写必填份量；
4. 自动换算营养；
5. 连续添加多个条目；
6. 确认整餐。

输入使用明确标签、单位和多行布局。数值输入初始为空，不显示无含义的 `0` 或 `00`。信息不完整时只能保留草稿或由用户主动请求 Agent 分析。

本阶段不实现食谱编辑、条码识别和图片识别。

### 6.3 训练录入 `/today/workout/new`

先创建训练场次，再添加力量或有氧项目。场次自动保存为服务器草稿，最终一次确认。力量动作可以从紧凑输入展开到逐组编辑；有氧提供运动、时长和可选设备消耗。

力量训练缺少时长时不显示消耗估算。固定程序估算值始终标为“约/估算”。

### 6.4 智能输入 `/today/smart-entry`

一段文本可解析多餐和多个训练条目：

1. 固定解析和本地目录匹配优先；
2. 唯一匹配直接生成确定性草稿；
3. 多个候选要求用户选择；
4. 未解析或不完整内容才允许调用 Agent；
5. 用户可编辑、删除或取消选择结果；
6. 选中记录以全有或全无事务确认。

### 6.5 编辑与新用户流程

已确认记录使用独立路由：

- `/today/meals/:mealId/edit`；
- `/today/workouts/:sessionId/edit`。

删除需要二次确认。编辑或删除成功后由后端重新计算当天汇总。

新用户必须完成以下流程后进入今日页：

1. 身体档案；
2. 总体目标；
3. 活动水平；
4. 每日目标计算或填写并确认；
5. 默认计划餐数。

## 7. API 与事务

接口统一位于 `/api/v1`。最终实现可根据现有 FastAPI 路由组织调整文件位置，但不得破坏以下职责边界。

### 7.1 档案与目标

- `GET /profile`
- `PUT /profile`
- `GET /goals`
- `PUT /goals/overall`
- `POST /targets/calculate`
- `POST /targets/agent-draft`
- `POST /targets/confirm`
- `GET /targets/history`

保存档案或总体目标时返回重算预览，不自动写入新的每日目标。

### 7.2 今日聚合

- `GET /days/{date}`
- `PATCH /days/{date}/meal-plan`
- `POST /days/{date}/meal-plan/set-default`

`GET /days/{date}` 一次返回目标快照、营养汇总、餐食和训练摘要，避免前端自行拼接并计算业务结果。

### 7.3 目录

- `GET /catalog/foods/search`
- `GET /catalog/exercises/search`
- `POST /catalog/foods/custom`
- `POST /catalog/exercises/custom`
- 独立的收藏和最近使用接口。

搜索顺序固定为：最近使用、收藏、用户私有目录、中文审核目录、其他开放数据。

### 7.4 草稿

- `POST /drafts`
- `GET /drafts/{id}`
- `PATCH /drafts/{id}`
- `POST /drafts/{id}/analyze`
- `POST /drafts/{id}/confirm`
- `DELETE /drafts/{id}`

自动保存携带版本号，旧版本覆盖新版本时返回 `409`。确认接口支持 `Idempotency-Key`，防止重复点击创建重复记录。

Agent 调用期间不持有数据库写事务。后端先获取并校验模型输出，再单独保存草稿。智能输入多记录确认使用单一事务，任一记录校验失败则全部回滚。

### 7.5 正式记录

餐食和训练分别提供读取、更新和删除接口。所有接口校验记录所有权。更新保存新的记录快照，不引用目录当前值；更新和删除在事务内触发当天汇总重算。

## 8. Agent 设计

### 8.1 能力

Agent 仅支持以下任务：

1. 根据用户已经选择的总体目标，对每日四项数值提供建议；
2. 对未知食物提供具体建议值、范围、份量和烹饪假设；
3. 推断训练类型、动作别名、主要/次要肌群和 MET 元数据；
4. 将自然语言拆分为多个餐食和训练候选草稿。

Agent 不得推断用户实际完成的组数、次数、重量、时长或设备消耗。缺少这些字段时必须要求用户补充。

### 8.2 结构化输出

每种 Agent 任务使用独立 JSON Schema 和独立版本化提示词。后端使用 Pydantic 校验结构化输出。草稿记录模型、提示词版本、调用状态、必要假设和 token 使用量。

Agent 输出仍须通过固定程序的字段和安全校验。模型配置缺失、密钥无效、超时、限流或输出不合法时：

- 保留草稿；
- 返回可恢复错误；
- 不生成伪造结果；
- 不影响表单、目录、公式和统计功能。

所有 Agent 调用均由用户主动触发。系统不自动测试模型，也不后台自动分析。

## 9. 目录数据与许可

运行时只查询本地 SQLite。外部 API 和下载文件只用于受控导入与更新。

采用分层目录：

1. 项目审核的中文高频食品和训练动作；
2. 开放全球数据；
3. 用户私有目录。

已调研的数据源和约束：

- [USDA FoodData Central](https://fdc.nal.usda.gov/api-guide)：CC0/公共领域食品数据；
- [Open Food Facts](https://openfoodfacts.github.io/openfoodfacts-server/api/)：ODbL，要求归属和数据库共享方式符合许可；
- [free-exercise-db](https://github.com/yuhonas/free-exercise-db)：Unlicense，提供公开训练动作；
- [wger](https://github.com/wger-project/wger)：代码为 AGPL，内容许可按条目保存并核验；
- [Compendium of Physical Activities](https://pacompendium.com/)：保留原始 MET 值和引用，不修改发布值。

每条公共目录记录保存来源、来源 ID、数据集版本、许可、归属、导入时间和原始值校验哈希。许可不兼容的数据保持来源边界，不重新打包成错误授权的统一数据集。

每个来源使用独立 Adapter：

```text
读取原始数据
-> 校验许可和版本
-> 映射统一中间模型
-> 标准化单位
-> 去重但保留来源
-> 建立别名和拼音
-> 写入来源分区
-> 重建 FTS5 索引
```

导入通过 `(source, source_record_id)` 唯一约束和数据集版本实现幂等。公共目录更新不修改历史记录快照。

## 10. 旧数据迁移

现有 CSV 自动迁移到 SQLite，要求：

- 迁移前创建带时间和校验哈希的只读备份；
- 迁移账本保证同一版本只成功执行一次；
- 转换在数据库事务中完成，失败回滚且不覆盖原 CSV；
- 旧餐食按原餐次标签映射；
- 每条旧训练记录独立创建场次，不猜测分组；
- 缺失字段保持未知，不调用 Agent 补全；
- 每条记录保存 `legacy_import` 和原始行标识；
- 成功后核对数量、日期范围和关键汇总，再切换到 SQLite。

## 11. 错误、安全与可恢复性

### 11.1 接口错误

- `422`：字段级业务校验错误；
- `409`：草稿版本冲突；
- `401/403`：登录失效或资源不属于当前用户；
- `429/503/504`：Agent 限流、不可用或超时。

确认失败不改变正式记录，草稿继续保留。前端显示保存中、已保存、保存失败、需要补充和可确认状态。网络中断时保留当前页面输入，恢复后再同步。

### 11.2 模型配置

- API 密钥由用户在设置页填写；
- 后端使用服务器固定主密钥进行静态加密；
- 主密钥不写入数据库或 Git；
- 读取接口不返回明文密钥；
- 日志、异常和跟踪数据禁止记录密钥；
- 只有用户点击“测试连接”才测试模型配置。

发送给模型的数据限制为当前任务所需的最小档案和输入内容。

### 11.3 可恢复性

- 草稿默认 30 天过期；
- 目录导入失败时继续使用上一个成功版本；
- CSV 迁移失败时不切换数据源；
- 数据库迁移、目录导入、公式和 Agent 提示词分别版本化。

结构化日志记录草稿状态、确认失败原因、公式版本、Agent 请求类型和耗时、目录导入结果以及 CSV 迁移核对结果，但不记录 API 密钥和不必要的用户原始正文。

## 12. 测试与验收

### 12.1 后端测试

- 每日目标公式、取整、安全下限和生效日期；
- 营养换算、差异提示和 MET 计算；
- 用户数据隔离；
- 草稿事务、幂等键和乐观锁冲突；
- 历史快照不受目录或目标新版本影响；
- 智能输入多记录全成功或全回滚；
- 无模型配置、超时、限流和非法结构输出的降级行为。

### 12.2 数据测试

- 每个 Adapter 使用固定小样本；
- 重复导入不产生重复记录；
- 来源、许可、版本和归属不可缺失；
- 搜索分层、中文、英文、别名和拼音匹配；
- CSV 备份、幂等、失败回滚和汇总核对。

### 12.3 前端测试

- 今日页不显示空餐，训练区域按存在性显示；
- 四项营养数据完整；
- 数值字段初始为空且具有标签和单位；
- 餐食多条目、力量逐组、有氧和草稿恢复；
- Agent 不可用时固定功能保持可用；
- 目标重算预览不自动覆盖；
- 草稿冲突和确认失败可恢复。

### 12.4 端到端验收

Playwright 覆盖：

1. 新用户档案和目标流程；
2. 搜索并确认一餐多条食物；
3. 自定义完整食物进入个人目录；
4. 未知食物经 Agent 分析后确认；
5. 力量和有氧混合场次；
6. 一段智能输入生成并原子确认多条记录；
7. 编辑历史记录后当天汇总更新；
8. 无模型配置时表单和固定计算正常；
9. Docker 环境启动与健康检查。

CI 使用模拟模型，不调用真实 API。真实模型只由用户在设置中手动测试。

## 13. 实施阶段

1. SQLite、Repository、数据库迁移和来源模型；
2. 档案、总体目标、固定计算、目标版本和 Agent 建议草稿；
3. 食品目录搜索、餐食草稿、确认和个人目录；
4. 动作目录、训练场次、逐组记录和消耗估算；
5. 今日聚合页、计划餐数、条件训练区和历史快照；
6. 确定性解析优先的智能输入与 Agent 回退；
7. 开放数据 Adapter、旧 CSV 自动迁移和许可文档；
8. 完整回归、Docker 验证、性能和错误恢复验收。

每个阶段必须形成包含前端、后端、固定程序和必要 Agent 边界的可验证纵向功能，避免最后一次性集成。

## 14. 明确不实施

本阶段不实施：

- 账户数据导出；
- Agent 总体目标推荐；
- Agent 直接写入、编辑或删除正式记录；
- 食谱组合器；
- 条码和图片识别；
- 运行时依赖外部食品或动作 API；
- 自动模型连接测试；
- PostgreSQL、搜索服务或异步任务队列；
- 完整审计管理后台。
